The Problem

Most VRP solvers are designed for static route planning.
Real dispatch is dynamic — and chaotic.

In production reality

  • Technicians are mid-route when changes hit
  • Drivers call in sick at 8 AM
  • Urgent SLA orders inject mid-day
  • Customers cancel after dispatch
  • Shifts get delayed or cut short
  • Service times shift as work expands

What standard solvers do

  • Re-solve from scratch every time
  • Reroute already-completed stops
  • Lose execution state
  • Miss SLA windows during re-planning
  • No awareness of in-progress work
  • Treat every change as a full restart

The Vision

Build a VRP solver that knows execution state — and a benchmark that proves it works.

1 · Compare

Side-by-side benchmark of 8 solver families on identical inputs.

Fair, reproducible, visual.

2 · Preserve

Lock in-progress routes so re-optimization respects committed work.

New: split-solve architecture.

3 · Self-correct

An AI agent that detects violations and triggers re-optimization autonomously.

The Split-Solve Innovation

Locked orders never enter the OR-Tools model

❌ Naïve approach

  • Submit all 150 orders to solver
  • 130 with zero-slack TW constraints
  • PATH_CHEAPEST_ARC fails to find feasible
  • 0 orders assigned
  • Solver wastes time on infeasibility
  • No way to improve from there

Split-solve

  • Submit only ~10 open orders
  • Locked stops copied verbatim from baseline
  • Vehicle anchored at last locked stop
  • GLS gets full 45s budget for improvement
  • 150/150 assigned · zero violations
  • Greedy fallback catches edge cases

Split-Solve Pipeline

1

Separate

Locked vs open orders by OrderId

2

Anchor

Each vehicle starts at last locked stop

3

Solve

Small OR-Tools model · full GLS budget

4

Merge

Locked verbatim + open from solver

5

Fallback

Greedy catch · 100% assignment

def _ortools_ea_solve_scenario(scenario, orders, routes, locked_prefixes):
    locked_ids = {oid for prefix in locked_prefixes.values() for oid in prefix.OrderId}
    open_orders = orders[~orders.OrderId.isin(locked_ids)]

    anchors = {r: _anchor_position(r, locked_prefixes[r]) for r in routes}

    solution = ortools_solve(open_orders, routes, anchors, time_limit=45)
    result = _merge_locked_with_solution(locked_prefixes, solution, anchors)

    return _greedy_catch_unassigned(result, open_orders)

Locked-Prefix Mechanism

Determines which stops are stable and can be frozen

Detection logic

  • changed_order_ids = baseline ⊕ scenario
  • impacted_routes = routes containing changes
  • fully_locked = routes with zero changes
  • partial_lock = stops up to first change

Special cases

  • RT_1 (urgent inject) → unlock 2 nearest routes
  • RT_6 (sick technician) → drop route entirely
  • RT_7 (combined) → multi-route unlock
  • Default → preserve all stable stops
def _prepare_locked_prefixes(scenario_key, baseline, scenario_orders):
    changed = _changed_order_ids(baseline.orders, scenario_orders)
    impacted = {r for r in baseline.routes
                if any(s.OrderId in changed for s in r.stops)}

    return {r.name: r.stops[:first_impact(r, changed)]
            if r.name in impacted else r.stops
            for r in baseline.routes}

The Agentic AI Layer

Powered by Google Gemini 2.0 Flash · free tier · 1,500 req/day

  • Wraps Gemini with function-calling
  • 8 callable tools (JSON Schema → FunctionDeclaration)
  • 4 distinct agentic behaviors
  • Max 10 iterations per query
  • Stateful chat session
  • Tool calls log for full audit trail

Why agentic, not just chat?

  • Detects violations autonomously
  • Picks config without human input
  • Re-runs solver, compares results
  • Chains multiple tool calls in one turn
  • No "ask permission" loops
  • Self-correcting under disruption

The Agent Loop

def _run_agent_loop(self, user_message: str) -> str:
    max_iterations = 10
    current_message = user_message

    while iteration < max_iterations:
        response = self._chat_session.send_message(current_message)

        # Collect function calls from response
        fn_calls = [p.function_call for c in response.candidates
                    for p in c.content.parts
                    if hasattr(p, "function_call") and p.function_call.name]

        if not fn_calls:  # Pure text — agent done
            return extract_text(response)

        # Execute tools and feed results back
        for fn_call in fn_calls:
            result = self._dispatch_tool(fn_call.name, dict(fn_call.args))
            self._tool_calls_log.append({"tool": fn_call.name, "result": result})

        current_message = build_function_response(fn_calls, results)
